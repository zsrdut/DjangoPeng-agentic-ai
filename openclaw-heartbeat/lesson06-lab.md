# 第6节 实验手册：心跳巡检与定时任务组合实战

> 配套课程：AI 业务流架构师 · 第6节 Heartbeat 心跳机制
> 前置条件：已完成第5节，Agent 已配置 SOUL.md 和 AGENTS.md
> 操作方式：全程在飞书中与 Agent 对话完成，无需登录服务器
> 预计耗时：实验一 ~45 分钟，实验二 ~30 分钟

---

## 实验目标

1. 理解 Cron（定时任务）和 Heartbeat（心跳巡检）的区别
2. 用 Cron + Heartbeat 组合架构搭建多轮资讯日报流
3. 用 Heartbeat 守护第5节写过的"每日记忆归档"规则
4. 验证心跳静默机制（HEARTBEAT_OK）的实际效果

## 实验一：Cron + Heartbeat 打造高质量日报

### 架构说明

```
07:00  [Cron Job 1]  → 独立 Session → 三领域资讯抓取 → 写入 briefing 文件
07:30  [Cron Job 2]  → 独立 Session → 读取第一轮结果 → 定向补充薄弱领域
08:00  [Heartbeat]   → 主 Session 巡检 → 检查日报是否完整 → 异常才推送飞书
```

Cron 负责"干活"（抓取数据），Heartbeat 负责"巡检"（检查质量）。

### 1a. 确认当前状态

在飞书中发送：

```
帮我检查一下当前的 Heartbeat 配置状态：
1. 读取 openclaw.json 中 heartbeat 相关的配置（every、target、activeHours 等）
2. 读取当前 HEARTBEAT.md 的内容（如果有的话）
3. 列出 memory/ 目录下现有的文件
4. 确认当前的时区设置
```

**确认要点**：heartbeat 已启用（`every` 不为 `0m`）、target 指向正确渠道、时区正确。

### 1b. 注册第一轮 Cron 任务（07:00）

在飞书中发送：

```
帮我创建一个 Cron 定时任务，具体要求如下：

- 任务名称：morning-briefing-round1
- 执行时间：每天早上 7:00（北京时间）
- Session 模式：isolated（独立 Session）
- 执行内容：使用 Web Search 分别搜索以下三个领域的最新资讯（过去24小时）：
  1. AI/大模型：重点关注模型发布、融资、开源项目
  2. 开源/技术：重点关注 GitHub Trending、重大版本发布
- 每个领域搜索 2-3 个关键词
- 汇总写入文件：memory/当天日期-briefing.md（YYYY-MM-DD 格式）
- 文件格式：按领域分 H2 标题，每条资讯包含标题、来源、一句话摘要
- 在文件末尾添加「## 覆盖评估」段落，标注哪些领域覆盖充分、哪些偏薄弱
- 完成后通过飞书发送通知

请创建这个 Cron 任务，并告诉我创建结果。
```

> 💡 **个性化提示**：三个领域是课程示例，你可以替换成自己关注的领域（如医疗/教育/跨境电商等）。

### 1c. 注册第二轮 Cron 任务（07:30）

在飞书中发送：

```
再创建一个 Cron 定时任务：

- 任务名称：morning-briefing-round2
- 执行时间：每天早上 7:30（北京时间）
- Session 模式：isolated（独立 Session）
- 执行内容：
  1. 先读取 memory/ 目录下今天的 briefing 文件
  2. 查看末尾的「覆盖评估」段落
  3. 针对标注为「偏薄弱」的领域，用不同的关键词角度做定向补充搜索
  4. 将补充结果追加到对应领域段落中，用「### 补充」子标题区分
  5. 更新覆盖评估
  6. 如果所有领域已覆盖充分，在文件末尾添加「✅ 二轮采集确认：覆盖完整」
  7. 如果找不到第一轮的 briefing 文件，发送告警
- 完成后通过飞书发送通知

请创建并确认。
```

确认两个任务都在：

```
列出所有 Cron 任务，确认两个 morning-briefing 任务都在
```

### 1d. 编写 HEARTBEAT.md 巡检项

在飞书中发送：

```
请帮我更新 HEARTBEAT.md 文件，用以下内容完整替换：

# Heartbeat 巡检清单

## 资讯日报完整性检查
- 在 08:00-09:00 时段内执行此项检查
- 检查 memory/ 目录下今天的日报文件 memory/YYYY-MM-DD-briefing.md 是否存在
- 如果文件不存在：发送告警「⚠️ 今日资讯日报未生成，请检查 Cron 任务状态」
- 如果文件存在但缺少「✅ 二轮采集确认」标记：发送提醒「资讯日报第二轮采集尚未完成」
- 如果文件存在且包含「✅ 二轮采集确认」：一切正常，回复 HEARTBEAT_OK

## 默认行为
- 以上检查项在对应时段外或均无异常时，回复 HEARTBEAT_OK
- 不要发送无意义的「一切正常」消息

写完后读回给我确认一下内容。
```

### 1e. 配置 activeHours

在飞书中发送：

```
请帮我检查 openclaw.json 中 heartbeat 的配置，确保包含以下设置：
- every: "30m"
- target: "last"
- activeHours: start "07:00"，end "23:00"

如果 activeHours 还没有配置，请帮我添加上去。时区要跟系统时区一致（Asia/Shanghai）。
修改后告诉我是否需要重启 gateway 才能生效。
```

### 1f. 手动触发验证

不等明天 07:00，现在手动跑一次：

```
请立即手动运行一次 morning-briefing-round1 这个 Cron 任务
```

等待 1-2 分钟，查看结果：

```
请读取今天的 briefing 文件，把内容发给我看看
```

> 🔍 **观察要点**：覆盖评估段落中是否有领域标注为"偏薄弱"——这正是第二轮存在的价值。

触发第二轮：

```
请立即手动运行一次 morning-briefing-round2
```

对比查看：

```
请再读一次今天的 briefing 文件，我要对比看看第二轮补充了什么
```

### 1g. 验证心跳巡检——三种场景

**场景 A：日报完整 → 期望静默**

```
请现在执行一次心跳巡检，检查资讯日报完整性
```

Agent 会在对话中回复检查结果（如"日报完整，无异常"）——这是正常的，因为对话式触发总会有回复。真正的静默需要等自然心跳周期触发。

**场景 B：日报缺失 → 期望告警**

```
请把今天的 briefing 文件临时重命名为 .bak，然后执行一次心跳巡检
```

你应该看到告警消息。验证后恢复文件：

```
请把 briefing 文件恢复原名
```

**场景 C：日报半完成 → 期望提醒**

```
请删掉 briefing 文件末尾的 ✅ 那行，然后执行心跳巡检
```

你应该看到提醒消息。验证后恢复：

```
请在 briefing 文件末尾重新加上「✅ 二轮采集确认：覆盖完整」
```

> ✅ 三种场景、三种不同结果——这就是 Heartbeat 的决策能力。

## 实验二：用 Heartbeat 敲打 Agent 遵守每日记录

### 2a. 翻车现场——记录断档实况

在飞书中发送：

```
请帮我做两件事：
1. 列出 memory/ 目录下所有文件（按日期排序），我想看看最近的日志情况
2. 找到 AGENTS.md 中关于「每日对话要点写入 memory/」的那条规则，把原文发给我
```

> 🔍 **观察要点**：规则白纸黑字写着，但 memory/ 目录可能已经断档多日——"规则写了但 Agent 没执行"。

追问 Agent：

```
你还记得这条规则吗——「每日对话要点写入 memory/YYYY-MM-DD.md」。你最近一次执行它是什么时候？为什么后来停了？
```

> 💡 Agent 的回答通常很有趣——它可能会承认"忘了"或分析出技术原因。无论哪种回答都说明一件事：**规则只是期望，机制才是保障。**

### 2b. 在 HEARTBEAT.md 追加记忆巡检项

在飞书中发送：

```
请在 HEARTBEAT.md 的末尾追加以下内容（保留之前已有的资讯日报检查部分）：

## 每日对话记忆归档检查
- 在 20:00-22:00 时段内执行此项检查
- 检查 memory/ 目录下今天的日志文件 memory/YYYY-MM-DD.md 是否存在（注意：不是 briefing 文件）
- 如果今天有过对话但日志文件不存在：
  1. 回顾今天的对话要点（关键决策、学到的信息、待办事项）
  2. 创建 memory/YYYY-MM-DD.md 文件并写入要点
  3. 发送通知「📝 已自动归档今日对话要点」
- 如果今天没有对话记录：回复 HEARTBEAT_OK
- 如果日志文件已存在：回复 HEARTBEAT_OK

追加完成后，读回完整的 HEARTBEAT.md 给我确认。
```

### 2c. 验证场景 A——有对话但没日志

先确认状态：

```
今天的日志文件 memory/今天日期.md 存在吗？帮我检查一下
```

Agent 应确认文件不存在（你今天聊了很多但从没写过日志）。触发巡检：

```
请现在执行一次心跳巡检，特别是记忆归档检查
```

你应该看到 Agent 回顾对话要点、创建日志文件、并通知你。查看内容：

```
请读取刚才自动创建的今天的日志文件，把内容发给我看看
```

> 🔍 **观察要点**：自动生成的对话要点是否准确？是否覆盖了今天的关键操作（Cron 任务创建、HEARTBEAT.md 编写等）？

### 2d. 验证场景 B——日志已存在

```
请再执行一次心跳巡检，检查记忆归档
```

Agent 会回复"日志已存在，无需操作"——因为刚才已经创建过了。真正的静默验证需要等自然心跳周期。

### 2e. 历史断档审计（可选）

```
请帮我做一个记忆完整性审计：检查最近 7 天中，哪些天有 daily log（YYYY-MM-DD.md），哪些天缺失。注意区分 daily log 和 briefing 文件，只看 daily log。不需要回填，告诉我缺失情况就行。
```
