# 飞书多维表格字段映射

生成出来的 `feishu_bitable_payload`，就是 transcript / LLM 理解结果与飞书多维表格写入之间的桥梁。

## 当前推荐的两表结构

### 1. 客户信息表

建议字段：
- 客户ID
- 客户名称
- 客户公司
- 行业
- MBTI
- 是否单身
- 沟通风格
- 成交阻力
- 价格敏感程度
- 风险顾虑
- 客户画像摘要
- 客户负责人
- 最后更新时间
- 数据来源

用途说明：
- 一位客户只保留一条主记录
- 每次会议后按 `客户ID` 做 upsert
- 用于沉淀长期客户画像，不适合记录每一轮推进历史

### 2. 商机快照表

建议字段：
- 商机ID
- 客户ID
- 客户名称
- 客户公司
- 机会名称
- 商机描述
- 当前阶段
- Lead Score
- 意向等级
- 高净值优先
- 销售区域
- 业务价值
- 推荐动作
- 最新进展
- 下次跟进时间
- 最近会议时间
- 商机负责人
- 数据来源

用途说明：
- 每次会议结束后 append 一条
- 不覆盖旧记录，保留完整推进轨迹
- 适合在飞书里做阶段看板、排序、过滤、提醒与复盘

## 字段写入建议

### 客户信息表写入策略
- 主键：`客户ID`
- 写入模式：`upsert`
- 更新内容：最新长期画像字段

### 商机推进快照表写入策略
- 主键：可不设唯一主键，按快照追加
- 写入模式：`append`
- 更新内容：本次会议结束后的商机状态切片

## 数据格式建议

- 数组字段统一拼成 `；` 分隔的中文字符串，便于在单元格内展示
- 时间字段统一写 ISO 8601 时间字符串
- 布尔字段可直接写 `true/false`，也可以在飞书侧映射为复选框
- `Lead Score` 保持数值型，便于排序和筛选
- `业务价值` 如果会话中未明确提到预算、金额或价值区间，建议统一写成 `暂无明确业务价值`

## 实践建议

- 不要把原始 transcript 全量写进主表
- transcript 原文、录音链接、会议纪要全文建议放在外部存储，或作为额外链接字段
- 真正落库前，可先让大模型或规则脚本输出标准 JSON，再由中间层转换成飞书 API 所需格式
- 如果后续需要扩展，可以在不改变两张主表的前提下，增加“任务提醒表”或“会议原文归档表”，但本 Skill 默认只输出两张核心表的写入载荷

## 字段对齐方式

如果你自己的飞书表字段名和项目默认字段名一致，可以直接写。  
如果不一致，建议在用户侧 Prompt 中明确补充：

- 你的客户信息表真实字段名
- 你的商机推进快照表真实字段名

规则是：
- 左边：项目输出字段名
- 右边：你飞书表里的真实字段名
