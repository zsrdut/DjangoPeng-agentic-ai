# Scope（P0）

## 1. 目标（Goals）
- 交付一个可复用的“财务识别 Skill”最小闭环：
  1) 文件输入（本地目录 / 飞书上传）
  2) OCR + 结构化字段抽取
  3) 合规检查与复核队列
  4) 飞书多维表格幂等写入
  5) 结果回执与运行可追踪

## 2. 范围（In Scope）
- 输入类型：`pdf` / `jpg` / `jpeg` / `png`
- 输入渠道：
  - 本地目录批处理
  - 飞书消息 webhook 文件上传
- 核心处理：
  - OCR（图片）与 PDF 文本提取
  - 发票字段抽取（金额、日期、发票号等）
  - 置信度与复核标记（`needs_review`）
  - 合规结果（`pass` / `warning` / `error`）
- 输出产物：
  - 结构化 JSON
  - review queue
  - compliance report
  - run summary
- Bitable：
  - 主表 + 复核表写入
  - 支持 dry-run
  - 幂等 upsert（避免重复入库）
- 飞书模式：
  - 单消息独立作业目录，避免混单

## 3. 非目标（Out of Scope）
- 老 ERP 浏览器自动填报
- 发票附件自动上传飞书云盘并回填附件列
- 复杂卡片审批流、多轮对话补录
- 模型训练与自动调优
- 企业级高可用部署（多实例编排、自动扩缩）

## 4. 完成定义（Definition of Done）
- 使用样例材料可稳定跑通端到端流程
- 低置信度或缺关键字段会进入复核队列
- Bitable dry-run 与真实写入都可执行
- 重复输入不会重复入库（幂等生效）
- 每次运行均有可追踪输出（summary / report / queue）

## 5. 约束与假设（Assumptions / Constraints）
- OCR 与抽取准确率受票据质量影响，P0 接受人工复核补位
- 密钥（飞书 app_id/app_secret/token）必须走环境变量
- 表字段命名需与同步映射一致
- 先保证“稳定可跑”，再追求“交互花哨”

## 6. 范围变更规则（Change Rule）
满足以下任一条件，视为超出 P0，进入 P1/P2 评审：
- 新增系统集成（如 ERP 自动填报）
- 新增关键数据链路（如附件自动上传）
- 新增复杂交互流（审批/卡片工作流）
- 影响当前稳定性的架构级改动
